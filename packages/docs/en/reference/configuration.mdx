## Config files

OpenFace creates and uses two files:

```text
~/.config/openface/config.json
~/.config/openface/model.json
```

- `config.json` stores user runtime overrides
- `model.json` stores metadata for pulled models

Both files are created automatically on first use.

## Merge behavior

The runtime config is merged with `defu`, so user settings extend the defaults instead of replacing the entire config.
This is similar to the layered config style used by tools such as OpenCode.

## Default cache location

Models are cached under:

```text
~/.local/share/openface/models
```

## Runtime defaults

The defaults are defined in `packages/openface/src/config/default.ts`. The important keys are:

- `LOG_LEVEL`
- `REMOTE_HOST`
- `CACHE_KEY`
- `CACHE_DIR`
- `LOCAL_MODEL_PATH`
- `REMOTE_PATH_TEMPLATE`
- `ALLOW_LOCAL_MODELS`
- `ALLOW_REMOTE_MODELS`

OpenFace explicitly sets the cache paths to `~/.local/share/openface/models` and inherits several other runtime values
from `@huggingface/transformers`.

## Example config

```json
{
  "CACHE_DIR": "/absolute/path/to/model-cache",
  "LOCAL_MODEL_PATH": "/absolute/path/to/model-cache",
  "ALLOW_REMOTE_MODELS": true
}
```

## Model registry format

Pulled models are recorded in `model.json` by provider and model name:

```json
{
  "provider": {
    "Helsinki-NLP": {
      "models": {
        "opus-mt-zh-en": {
          "id": "Helsinki-NLP/opus-mt-zh-en",
          "task": "translation"
        }
      }
    }
  }
}
```

`run` depends on this registry. If a model exists in the cache directory but is missing from `model.json`, the CLI will
not treat it as available.

## Inspecting config

Use the built-in commands to inspect the merged runtime state:

```bash
openface config list
openface config get cache
```

`config get <name>` performs a case-insensitive substring match against the merged config keys.
